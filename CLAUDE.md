# AutoTex Book Generator

Agent-based book generator in LaTeX. Claude Code is the primary agent — it plans, writes chapters, reviews formatting, and fixes issues. The Python package (`autotex/`) provides utility commands for compilation, advanced image generation (Gemini), and PDF rendering.

Every book is **KDP-compliant by default** — correct trim sizes, margins, and gutter for print-on-demand binding. The layout uses oneside geometry with gutter on the left (binding) margin. The focus is on professional, elegant formatting and general coherence.

## Project Structure

```
config.yaml          - Book metadata, KDP settings, Gemini config
main.tex             - Root LaTeX document (generated by `autotex update-main`)
autotex-book.sty     - Custom style: KDP page sizes, professional typography (no default boxes)
text/chapterNN.tex   - Individual chapter files (you write these)
assets/              - Generated images (created by `autotex images`)
build/               - Compilation output (PDF, aux files, rendered PNGs)
autotex/             - Python utility package
```

## CLI Commands

All commands: `python -m autotex <command>`

| Command | Description |
|---------|-------------|
| `init <name>` | Create a new book project in `books/<name>/` |
| `compile` | Compile main.tex to PDF via pdflatex (2 passes) |
| `render <pages...>` | Render PDF pages to PNG (0-based page numbers) |
| `render-chapter <num> <title>` | Render last N pages of a specific chapter |
| `images <file.tex>` | Extract image markers, generate via Gemini, replace with \includegraphics |
| `search <text>` | Find pages containing text in the compiled PDF |
| `update-main` | Regenerate main.tex from config.yaml chapters and KDP settings |
| `info` | Print current project configuration |

For book-specific commands, use `--config` to target a book subfolder:
```
python -m autotex --config books/<name>/config.yaml <command>
```

## Book Subfolder Workflow

Each book lives in its own self-contained subfolder under `books/`. The root project is the engine/template — never modify root files when working on a book.

### Creating a new book
```bash
python -m autotex init my-book
```
This creates `books/my-book/` with `config.yaml`, `CLAUDE.md`, `text/`, `assets/`, `build/`.

### Working on a book
All CLI commands use `--config` to target the book:
```bash
python -m autotex --config books/my-book/config.yaml update-main
python -m autotex --config books/my-book/config.yaml compile
python -m autotex --config books/my-book/config.yaml render-chapter 1 "Introduction"
python -m autotex --config books/my-book/config.yaml images text/chapter01.tex
```

### Book folder structure
```
books/my-book/
├── config.yaml    - Book metadata, KDP settings (edit this for planning)
├── CLAUDE.md      - Book-specific instructions (topic, audience, style)
├── main.tex       - Generated by update-main
├── text/          - Chapter .tex files
├── assets/        - Generated images
└── build/         - Compilation output (PDF, PNGs)
```

### Key rules
- **Never edit root-level** `config.yaml`, `main.tex`, `text/`, or `assets/` when working on a book
- Edit the **book's** `CLAUDE.md` for book-specific behavior (topic, tone, audience)
- `autotex-book.sty` stays at the repo root — the compiler finds it automatically via TEXINPUTS

## KDP Compliance

All books are KDP-compliant by default:
- **Trim sizes**: pocket (5.25x8), digest (5.5x8.5), novel (6x9, default), royal (6.14x9.21), crown (7x10), large (7.5x9.25), letter (8.5x11), ustrade, uktrade
- **Gutter**: 0.375in default (binding margin), configurable via `gutter` in config
- **Bleed**: optional 0.125in bleed for full-bleed images
- **Margins**: left (binding side) = outer + gutter, comfortable defaults above KDP minimums
- **Layout**: oneside geometry, gutter added to the left margin for binding

To change the trim size, edit the book's `config.yaml`:
```yaml
kdp:
  enabled: true
  trim_size: "novel"    # or pocket, novel, royal, crown, large, letter
```

## Book Planning

When asked to plan a book:
1. **Ask the user** the following questions before doing anything else (wait for their answers):
   - **Target audience**: Who is this book for? (age, background, expertise level)
   - **Trim size**: Which KDP size? (pocket 5.25×8, digest 5.5×8.5, novel 6×9, royal 6.14×9.21, crown 7×10, large 7.5×9.25, letter 8.5×11) — default is novel
   - **Length**: Roughly how many chapters / pages?
   - **Color style**: What mood/palette for the box palette and cover? (e.g., minimal black & white, warm earth tones, vivid academic blues, etc.)
   - **Tone & style**: Formal textbook, conversational guide, narrative, reference manual?
   - **Illustrations**: Prefer TikZ diagrams, AI-generated images, or both?
2. Run `python -m autotex init <book-name>` if the book subfolder doesn't exist yet
2. Read the book's `config.yaml` for existing settings
3. Create a structured plan with chapters, each having: number, title, sections, description
4. Build a logical progression from fundamentals to advanced topics
5. Include an introductory chapter and a concluding chapter
6. Update the book's `config.yaml` — set `book.title` and `book.chapters` array
7. Run `python -m autotex --config books/<name>/config.yaml update-main` to regenerate main.tex
8. Design the book's custom box palette and add it to `main.tex` (see **Custom Box Palette** section)

## Writing LaTeX Chapters

### Guidelines
- Use `\chapter{Title}` and `\label{ch:slug}` at the start
- Use `\section{}`, `\subsection{}`, `\subsubsection{}` for structure
- Always add `\label{}` to chapters, sections, figures, tables, and equations
- Use `\cref{}` for cross-references (e.g., `\cref{fig:example}`)
- Use booktabs commands (`\toprule`, `\midrule`, `\bottomrule`) for tables
- Use the book's custom boxes to highlight key concepts — see the box palette defined in `main.tex`
- For advanced and reliable AI-generated images, use the `%%IMAGE_REQUEST{...}%%` marker
- Keep paragraphs focused and well-structured
- Include mathematical notation where appropriate
- Do NOT include `\documentclass`, `\usepackage`, `\begin{document}`, or `\end{document}` — only write the chapter content starting with `\chapter{}`

### Custom Box Palette (MANDATORY for every book)

**Before writing any chapter**, define the book's custom tcolorbox environments in `main.tex`, after `\usepackage{autotex-book}`. There are NO default boxes — you must design a coherent palette from scratch for every book.

**Rules:**
- Design the palette to match the book's **theme, genre, and audience** (e.g., a mathematics textbook calls for a different aesthetic than a persuasion guide or a fantasy worldbuilding book).
- Choose **3-5 semantic roles** appropriate to the content (e.g., definition/theorem/example/note for academic books; principle/exercise/insight/warning for practical guides; lore/mechanic/example/tip for game design books).
- Name the environments descriptively to match their role in this specific book.
- Styles must be **elegant and intentional**: coherent color palette, consistent spacing, purposeful use of rules, backgrounds, and typography. Avoid generic rainbow-colored boxes.
- All environments must feel like a unified design system, not independent choices.

Add the palette definitions to `main.tex` in a clearly delimited block:
```latex
% ── Book box palette ────────────────────────────────────────────────
% (custom tcolorbox environments for this book)
\newtcolorbox{...}[1][]{...}
% ────────────────────────────────────────────────────────────────────
```

### Available Environments

1. **Figures** (images are in the `assets/` directory):
   ```latex
   \begin{figure}[htbp]
     \centering
     \includegraphics[width=0.8\textwidth]{filename}
     \caption{Caption text.}
     \label{fig:label}
   \end{figure}
   ```

2. **Tables** (use booktabs):
   ```latex
   \begin{table}[htbp]
     \centering
     \begin{tabular}{lcc}
       \toprule
       Header 1 & Header 2 & Header 3 \\
       \midrule
       Data & Data & Data \\
       \bottomrule
     \end{tabular}
     \caption{Caption text.}
     \label{tab:label}
   \end{table}
   ```

3. **Code listings:**
   ```latex
   \begin{lstlisting}[language=Python, caption=Caption]
     code here
   \end{lstlisting}
   ```

4. **TikZ diagrams:**
   ```latex
   \begin{tikzpicture}
     ...
   \end{tikzpicture}
   ```

5. **PGFPlots graphs:**
   ```latex
   \begin{tikzpicture}
     \begin{axis}[xlabel=X, ylabel=Y, title=Title]
       \addplot coordinates {(0,0) (1,1) (2,4)};
     \end{axis}
   \end{tikzpicture}
   ```

6. **Math** (amsmath, amssymb, mathtools are loaded):
    ```latex
    Inline: $f(x) = x^2$
    Display: \[ \int_0^\infty e^{-x} \, dx = 1 \]
    Aligned:
    \begin{align}
      a &= b + c \\
      d &= e + f
    \end{align}
    ```

## Illustrations Strategy

**TikZ/PGFPlots first, AI images else.** Prefer native LaTeX illustrations:

### Decision Tree

1. **Graphs, plots, function curves** → PGFPlots (native, pixel-perfect, scalable)
2. **Geometric diagrams, coordinate systems, simple schematics** → TikZ (native, precise)
3. **Flowcharts, block diagrams, trees** → TikZ with `shapes`, `arrows.meta` libraries
4. **Conceptual illustrations, realistic scenes, complex visual metaphors, anything else** → AI image (Gemini)

Only use AI-generated images when the illustration requires visual richness that TikZ cannot reasonably produce (e.g., 3D renderings, realistic textures, artistic interpretations).

### TikZ Best Practices

- Wrap TikZ in `\begin{figure}[htbp]...\end{figure}` with `\caption` and `\label`
- Use `>=stealth` or `>=latex` for clean arrow tips
- Keep diagrams simple: fewer elements = cleaner output
- Use consistent color scheme: match the tcolorbox palette (blue, green, orange, red)
- Test compilation after each diagram — TikZ errors are cryptic

### AI Image Request Markers (Gemini)

Place this marker on its own line:
```
%%IMAGE_REQUEST{description="...", filename="chNN-descriptive-name", aspect_ratio="16:9"}%%
```

After running `python -m autotex images text/chapterNN.tex`, the marker is replaced with a `\begin{figure}...\includegraphics...\end{figure}` block.

**Critical rules for AI image descriptions:**
- **Keep it simple.** Request ONE clear concept per image, not complex multi-element scenes.
- **Be extremely precise.** Describe exact positions, colors, labels, and relationships. Vague prompts produce vague results.
-  **Text rendering**: If you want a short text in the image, enclose it in quotes within the description:
   ```
   %%IMAGE_REQUEST{description="A labeled diagram with the text "Mitochondria" pointing to the organelle", ...}%%
   ```
- **Prefer schematic over realistic.** Flat, diagrammatic styles produce more consistent results than photorealistic requests.
- **Character/subject consistency**: (Optional) upload a reference image of the subject to maintain visual features across multiple images. Set the first generated image as the reference for subsequent ones.



### Supported Aspect Ratios

| Ratio | Use case | LaTeX width |
|-------|----------|-------------|
| `16:9` | Wide diagrams, flowcharts (default) | 0.85\textwidth |
| `3:2` | Standard landscape illustrations | 0.8\textwidth |
| `4:3` | Moderate landscape, screenshots | 0.75\textwidth |
| `1:1` | Square diagrams, icons | 0.6\textwidth |
| `3:4` | Moderate portrait, tall diagrams | 0.5\textwidth |
| `2:3` | Portrait illustrations | 0.45\textwidth |
| `9:16` | Tall narrow diagrams | 0.4\textwidth |

### Reference Images for Style Consistency

Add a `reference` field pointing to a file in `assets/`:
```
%%IMAGE_REQUEST{description="...", filename="ch02-cell-diagram", aspect_ratio="4:3", reference="assets/ch01-cell-overview.png"}%%
```

- Gemini matches the reference's color palette, line weight, and overall aesthetic
- Use this for style consistency across a chapter or to request edits to an existing image
- Explicitly include in your prompt what to do with the reference picture

## Chapter Generation Workflow

Set `CFG=books/<name>/config.yaml` for the active book.

**IMPORTANT: Write and review ONE chapter at a time. NEVER batch multiple chapters before reviewing.**

For each chapter:
1. Read the plan from the book's `config.yaml`
2. Read previous chapters (`books/<name>/text/chapter01.tex`, etc.) for context and continuity
3. Write the chapter content to `books/<name>/text/chapterNN.tex`
4. Run: `python -m autotex --config $CFG images text/chapterNN.tex` (generates images and replaces markers)
5. Run: `python -m autotex --config $CFG compile`
6. If compilation fails, read error output, fix the LaTeX, recompile (up to 3 attempts)
7. **Log analysis (MANDATORY):** Read `build/main.log` and count Overfull/Underfull boxes. If any Overfull hbox exceeds 10pt, you MUST fix it before proceeding.
8. **Full chapter render (MANDATORY):** Determine the page range of the chapter (use `autotex search "Chapter Title"` to find the start page, and the next chapter's start to find the end). Then render ALL pages of the chapter using `autotex render <start> <start+1> ... <end>`.
9. **Read EVERY rendered PNG.** Do NOT skip any page. Do NOT sample. Read them ALL, in order. For each page, check the Visual Review Checklist below.
10. If ANY issue is found, edit the LaTeX, recompile, and re-render the affected pages. Repeat up to 3 times.
11. Only after all pages pass review, move to the next chapter.

For very long books (15+ chapters), read only the most recent 2-3 chapters for context continuity rather than all of them.

### Final Book Review (after all chapters are written)

After completing ALL chapters, perform a final full-book review:
1. Render ALL pages of the entire PDF (page 0 through last page).
2. Read EVERY page PNG. Do NOT skip any.
3. Pay special attention to: title page, table of contents, part pages, chapter openings, and the last page of each chapter (where blank pages tend to appear).
4. Fix any remaining issues, recompile, and re-check affected pages.

## Visual Review Checklist

**You MUST check EVERY rendered page for ALL of the following. Do not skip items.**

1. **Header overlap**: Do the left header (chapter title) and right header (section title) collide or overlap? If the page is too narrow for both, the text will run together in the middle.
2. **Text overflowing into margins** (overfull hbox): Look for text that extends past the right margin. This appears as text jutting out beyond the text block boundary.
3. **Blank or near-blank pages**: Does the page have mostly white space with only a header and page number? This wastes space and looks unprofessional. Common causes: `\include` forcing new pages, `openright` creating blank even pages, or floats pushing content to the next page.
4. **Large vertical gaps**: Is there a gap of more than ~3cm between two elements (boxes, paragraphs, figures)? This usually means a float was placed poorly or a page break was forced.
5. **Bad figure/table placement**: Is a figure or table floating far from where it's referenced in the text? It should appear on the same page or the next page.
6. **Orphan/widow lines**: A single line of a paragraph stranded at the top or bottom of a page.
7. **Overlapping elements**: Text overlapping with figures, tables, or boxes.
8. **Missing cross-references**: Any "??" in the text means a `\cref{}` or `\ref{}` is broken.
9. **Consistent styling**: Fonts, spacing, and heading styles should be uniform throughout.
